== Description

=== Impact
The case is, in https://github.com/azure-rtos/usbx/blob/master/common/usbx_host_classes/src/ux_host_class_pima_read.c[_ux_host_class_pima_read], there is data length from device response, returned in the very first packet, and read by https://github.com/azure-rtos/usbx/blob/082fd9db09a3669eca3358f10b8837a5c1635c0b/common/usbx_host_classes/src/ux_host_class_pima_read.c#L165[L165 code], as header_length. Then in https://github.com/azure-rtos/usbx/blob/082fd9db09a3669eca3358f10b8837a5c1635c0b/common/usbx_host_classes/src/ux_host_class_pima_read.c#L178[L178 code], there is a “if” branch, which check the expression of “(header_length - UX_HOST_CLASS_PIMA_DATA_HEADER_SIZE) > data_length”  where if header_length is smaller than UX_HOST_CLASS_PIMA_DATA_HEADER_SIZE, calculation could overflow and then https://github.com/azure-rtos/usbx/blob/082fd9db09a3669eca3358f10b8837a5c1635c0b/common/usbx_host_classes/src/ux_host_class_pima_read.c#L182[L182 code] the calculation of data_length is also overflow, this way the later https://github.com/azure-rtos/usbx/blob/082fd9db09a3669eca3358f10b8837a5c1635c0b/common/usbx_host_classes/src/ux_host_class_pima_read.c#L192[while loop start from L192] can move data_pointer to unexpected address and cause write buffer overflow.

=== Patches

We analyzed this bug and determined that we needed to fix it. This fix has been included in USBX release https://github.com/azure-rtos/usbx/releases/tag/v6.1.12_rel[6.1.12]

=== Workarounds

Add check of `header_length`:
1\. It must be greater than `UX_HOST_CLASS_PIMA_DATA_HEADER_SIZE`.
1\. It should be greater or equal to the current returned data length (`transfer_request -> ux_transfer_request_actual_length`).
